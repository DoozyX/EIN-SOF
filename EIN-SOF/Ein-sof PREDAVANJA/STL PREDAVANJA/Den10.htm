Класата auto_ptr (smart pointer) и динамичка алокација на меморија

C++  стандардот предвидува class template auto_ptr имплементиран во <memory>. Објект од оваа класа содржи покажувач на динамички алоцирана меморија. Кога овој објект ќе излезе од областа на постоење, тој врши delete на покажувачот. auto_ptr обезбедува оператори * и ->, па може да се користи како стандарден покажувач. Следниот пример демонстрира употреба на auto_ptr: 

 

 

// Demonstacija na auto_ptr.

#include <iostream>

 

using std::cout;

using std::endl;

 

#include <memory>

 

using std::auto_ptr; // definicija na klasata auto_ptr

 

class Integer {

public:

 

   // Integer konstruktor

   Integer( int i = 0 )

      : value( i )

   {

      cout << "Konstruktor za Integer " << value << endl;

 

   }

 

   // Integer destruktor

   ~Integer()

   {

      cout << "Destruktor za Integer " << value << endl;

 

   }

    void setInteger( int i )

   {

      value = i;

 

   }

 

   int getInteger() const

   {

      return value;

 

   }

 

private:

   int value;

 

};  // kraj na klasa Integer

 

// upotreba na auto_ptr

int main()

{

   cout << "Kreiranje na auto_ptr objekt sto "

        << "pokazuva naInteger\n";

 

   // postavi auto_ptr

   auto_ptr< Integer > ptrToInteger( new Integer( 7 ) );

 

   cout << "\nUpotrebuvam auto_ptr za manipuliranje so Integer\n";

 

  

   ptrToInteger->setInteger( 99 );

 

  

   cout << "Integer po setInteger: "

        << ( *ptrToInteger ).getInteger()

        << "\n\n\kraj na programata" << endl;

 

   return 0;

 

}  // kraj na main

 

Излезот е:

Kreiranje na auto_ptr objekt sto pokazuva naInteger

Konstruktor za Integer 7

 

Upotrebuvam auto_ptr za manipuliranje so Integer

Integer po setInteger: 99

 

kraj na programata

Destruktor za Integer 99

Кога main ќе заврши, и ptrToInteger се уништува бидејќи е автоматска променлива. Со тоа се уништува и Integer објектот со повикување на неговиот деструктор. Уништувањето на Integer настанува без разлика дали програмата завршува нормално или се јавува исклучок. На тој начин се спречува појава на  memory leak. Постојат ограничувања на употреба на auto_ptr. Тој не може да покажува на поле ниту на стандарден контејнер.

 Стандардни исклучоци

Во стандардната библиотека на C++ постојат стандардни исклучоци. Можат да се користат објекти од тие класи, а може и да се наследуваат за создавање на кориснички класи со исклучоци.

Сите стандардни класи наследуваат од класата exception, која е дефинирана во <exception>. Од неа се изведени две главни класи logic_error и runtime_error, кои се дефинирани во <stdexcept> (која исто така ја вклучува <exception>). Со logic_error се претставуваат грешки во програмската логика, како на пример пренесување на несоодветен аргумент. Со runtime_error се претставуваат грешки што настануваат при извршување, како на пример недостатoк на меморија. Двете класи имаат конструктор со аргумент std::string во кој се сместува пораката за грешка во објектот. Таа порака се враќа преку членката exception::what( ), што е илустрирано во следниот пример:

//Nasleduvanje od std::runtime_error.

#include <stdexcept>

#include <iostream>

using namespace std;

 

class MyError : public runtime_error {

public:

  MyError(const string& msg = "") : runtime_error(msg) {}

};

 

int main() {

  try {

    throw MyError("Moja poraka");

  } catch(MyError& x) {

    cout << x.what() << endl;

  }

} 

 

Иако конструкторот на runtime_error вметнува порака во std::exception, std::exception нема конструктор со std::string аргумент. Затоа најчесто се наследува од runtime_error или logic_error, а не директно од std::exception.

Следната табела ги содржи стандардните класи на исклучоци:

exception

Основна класа

logic_error

Изведена од exception. Пријавува програмски грешки

runtime_error

Изведена од exception. Пријавува грешки при извршување

 Класата ios::failure е изведена од exception, но нема други класи изведени од неа.

Класи изведени од logic_error

domain_error

Пријавува нарушување на предуслови

invalid_argument

Пријавува невалиден аргумент на функција од која е фрлен исклучок.

length_error

Се пријавува при обид да се создаде објект со големина поголема од максимално дозволената

out_of_range

Пријавува out-of-range грешка.

bad_cast

Се фрла при извршивање на dynamic_cast израз

bad_typeid

Пријавува null pointer p во израз typeid(*p).

 

Класи изведени од runtime_error

range_error

Пријавува нарушување на ранг

overflow_error

Пријавува аритметичко преполнување

bad_alloc

Пријавува неуспех при алоцирање на простор

 

Спецификации на исклучок

 

 

Спецификација на исклучок (exception specification, throw list) е листа на исклучоци кои една функција може да ги фрли. Оваа листа започнува со клучниот збор throw како во следниот пример:

 

int Function( double value)

throw(ExceptionA, ExceptionB, ExceptionC)

{

//telo na funkcijata

}

 

Функцијата може да фрла само исклучоци од типовите во листата кои можат да бидат објекти и изрази чии резултати се од наведените типови. Ако функцијата фрли исклучок од тип кој не припаѓа на листата, тогаш се извршува функцијата unexpected.

Стандардна декларација на функција:

void f();

 

значи дека било кој тип на исклучок може да биде фрлен од неа. Декларацијата:

void f() throw();

значи дека никаков исклучок нема да биде фрлен од офункцијата f.

Ако функција фрли исклучок што не е во нејзината спецификација на исклучоци се повикува функцијата unexpected( ). Предефинираното однесување на оваа функција е повик на функцијата terminate( ).

 Промена на ова однесување на unexpected( ) може да се изврши преку функцијата set_unexpected( ), која како аргумент зема адреса на функција без аргументи која има повратна вредност void. set_unexpected( ) враќа покажувач на последната функција која била повикана од  terminate( ) и unexpected( ) За да се повика set_unexpected( ), треба да се вклучи <exception>. Слично, промена на однесувањето на terminate( ) може да се изврши преку функцијата set_terminate( ).

Следниот пример илустрира употреба на set_unexpected( ):

#include <exception>

#include <iostream>

using namespace std;

 

class Up {};

class Fit {};

 

void g();

 

void f(int i) throw(Up, Fit) {

  switch(i) {

    case 1: throw Up();

    case 2: throw Fit();

  }

  g();

}

 

//void g() {}         // verzija 1

void g() { throw 50; } // verzija 2

 

void my_unexpected() {

  cout << "unexpected exception thrown" << endl;

  exit(0);

}

 

int main() {

  set_unexpected(my_unexpected);

  for(int i = 1; i <=3; i++)

    try {

      f(i);

    } catch(Up) {

      cout << "Up faten" << endl;

    } catch(Fit) {

      cout << "Fit faten" << endl;

    }

 

}

Пишување на спецификација на исклучоци за функции кои фрлаат исклучоци може да ги подобрат читливоста и документираноста на кодот. Но, ако однапред не се познати сите исклучоци кои можат да настанат, тогаш пишување на спецификација не е препорачливо. Особено не кај template класи и функции.

 

Испразнување на стекот за функциски повици (stack unwinding)

 

 Кога исклучок е фрлен, но не е фатен, се врши празнење на стекот за функциски повици и се прави обид да се фати исклучокот со следниот надворешен try/catch блок. Празнење на стекот за функциски повици значи дека се прекинува функцијата во која исклучокот не е фатен, сите нејзини локални променливи се уништуваат и контролата се пренесува кај наредбата која ја повикала таа функција. Ако околу таа наредба има try блок, се прави обид да се фати исклучокот. Ако нема, повторно се празни стекот за тој блок. Ако не дојде до фаќање на исклучокот, се повикува terminate().

 

Следниот пример илустрира празнење на стекот за функциски повици.

 

 

//  stack unwinding.

#include <iostream>

 

using std::cout;

using std::endl;

 

#include <stdexcept>

 

using std::runtime_error;

 

// function3 frla run-time error

void function3() throw ( runtime_error )

{

   throw runtime_error( "runtime_error vo function3" );

}

 

 

void function2() throw ( runtime_error )

{

   function3();

}

 

 

void function1() throw ( runtime_error )

{

   function2();

}

 

int main()

{

  

   try {

      function1();

 

   }

 

  

   catch ( runtime_error &error )

   {

      cout << "Nastanal isklucok: " << error.what() << endl;

 

   }

 

   return 0;

 

} 

 

 
Излезот е:
 

Nastanal isklucok: runtime_error vo function3

 

try блокот во main ја повикува function1, таа ја повикува function2, а  function2 ја повикува function3 која фрла исклучок од тип runtime_error. Бидејќи нема try блок околу function3, доаѓа до празнење на стекот. Контролата се предава на function2, повторно нема try блок околу неа и повторно се празни стекот и контролата се предава погоре. Процесот продолжува додека не дојде до main, каде се процесира исклучокот.

 

Конструктори, деструктори и exception handling

 

Конструкторите не можат да враќаат вредности, па не можат да го користат тој начин за известување на настаната грешка при создавање на објект. Еден начин е да се врати објектот кој не е прописно креиран и да се очекува дека вредноста ќе биде проверена кај повикувачката функција. Друга алтернатива е да се постави надворешна променлива. Можеби најдобро решение е механизмот на exception handling.

Кога конструктор ќе фрли исклучок, тоа ќе предизвика да се повикаат деструктори за сите објекти што биле создадени рамки на конструкторот пред да биде фрлен исклучокот. Деструктори ќе бидат повикани и за сите автоматски објекти во try блок пред да биде фрлен исклучок. Празнењето на функцискиот стек сигурно ќе биде завршено пред да почне да се извршува exception handler-от. Ако деструктор, кој е повикан како резултат на празнење на функцискиот стек, фрли исклучок тогаш се повикува terminate(). Затоа не треба да има фрлање на исклучок во деструктор.

Ако настане исклучок при креирање на поле, тогаш ќе бидат повикани деструктори само за елементите што биле успешно креирани.

 

Исклучоци и наследување

 

Исклучоците можат да се изведуваат од заедничка класа. Ако exception handler фаќа референца на основна класа, може да фаќа и исклучоци од објекти од сите класи изведени од основната. Подобро е да се користи exception handler кој фаќа исклучок од основна класа, отколку да се пишува exception handler за секоја изведена класа поединечно.

 

Процесирање на грешки настанати при new

 

Според најновиот C++ стандард, кога ќе настане грешка при new, се фрла исклучок bad_alloc кој е дефиниран во <new>. Некои компајлери се компатибилни со стандардот, но некои сеуште импламентираат верзија на new која враќа 0 при грешка.

Следниот пример прикажува new која враќа 0 при грешка:

 

 

//  new vrakja 0 koga ne e alocirana memorija

 

#include <iostream>

 

using std::cout;

 

int main()

{

   double *ptr[ 50 ];

 

   // alociraj memorija za ptr

   for ( int i = 0; i < 50; i++ ) {

      ptr[ i ] = new double[ 5000000 ];

 

      // new vrakja 0 pri neuspeh

      if ( ptr[ i ] == 0 ) {

               cout << "Alociranje na memorija neuspesno za ptr[ " << i << " ]\n";

 

         break;

 

      }

 

     

      else

         cout << "Alocirani 5000000 double elementi ptr[ "

              << i << " ]\n";

 

   }

 for(int j=0; j<i;j++)

         delete ptr[j];

 

   return 0;

 

}  // kraj na main

for(int j=0; j<i;j++)

НАПОМЕНА: Обработката на неуспех при new зависи од компајлерот. Кај многу од нив се враќа 0 при неуспех, некои фрлаат bad_alloc по default, a некои поддржуваат фрлање исклучок ако е вклучен <new>. Ако new не фрла исклучок кога нема доволно меморија туку враќа 0 како резултат, програмата нема да го даде истиот излез.

 

Со set_new_handler() може да се постави корисничка функција која ќе се повика при неуспех на new. Ако е поставена ваква функција, тогаш не се фрла исклучок и се повикува таа функција при неуспех на new.

 Стандардот пропишува дека оваа функција треба да извршува една од следните задачи:

  • Создавање на повеќе меморија преку ослободување на друга динамички алоцирана меморија и да ја предаде контролата на new за повторен обид за алокација 
  • Да фрли исклучок од тип bad_alloc
  • Да повика abort() или exit() за да ја прекине програмата.

 

Вежби:

 

1.      Напиши програма која демонстрира дека ако е фрлен покажувач на создаден објект, објектот нема да биде уништен.

2.      Креирај класа Garaza во која има Avtomobil што може да има проблеми со својот Motor. Употреби try блок во конструкторот на Garaza за фаќање на исклучокот од Motor кога се иницијализира Avtomobil. Фрли друг исклучок од catch блокот во конструкторот на  Garaza и фати го во main( ).
